# Known Issues — повторяющиеся проблемы и их решения

## Purpose

Документирует типичные ошибки, которые повторяются при разработке CodePatrol. Каждая проблема описана с контекстом, решением и ссылкой на то, как она закрыта в текущем коде.

## When to read

- Перед изменением шаблонов скиллов (`templates/`)
- Перед изменением build pipeline (`install.sh`, `platforms/*.env`)
- При добавлении нового скилла
- При изменении workflow между скиллами (cp-review → cp-fix, brainstorming → writing-plans)

---

## 1. Забываем пересобрать skills/ после изменения templates/

**Симптом:** Изменения в шаблонах не отражаются при использовании скиллов. Тестируешь — работает старая версия.

**Причина:** `skills/` — сгенерированная директория. Прямое редактирование `skills/` перезатрётся при следующей сборке, а изменение `templates/` без сборки не попадёт в `skills/`.

**Решение в коде:** Правило в `CLAUDE.md`:
> After editing any file in `templates/`, run `./install.sh build` to regenerate `skills/`

**Чеклист:**
- [ ] После любого изменения в `templates/` — `./install.sh build`
- [ ] После изменения `platforms/*.env` — `./install.sh build`
- [ ] После изменения `templates/_shared/` — `./install.sh build` (затрагивает все скиллы, которые включают этот partial)

---

## 2. CI/CD падает из-за рассинхрона версий

**Симптом:** Release workflow на GitHub падает. Причины: несовпадение версий, невалидный manifest, несуществующие пути.

**Причина:** Ручное обновление версии в нескольких местах, разные файлы ожидают разные форматы.

**Решение в коде:** CI-валидация проверяет совпадение версий в plugin.json и marketplace.json, наличие всех заявленных skill-файлов, валидность JSON манифестов.

**Чеклист перед релизом:**
- [ ] Версия обновлена в `plugin.json` и `marketplace.json`
- [ ] `./install.sh build` выполнен
- [ ] Локальная валидация пройдена
- [ ] Коммит на dev → merge в main → tag → release

---

## 3. Codex SKILL.md — невалидный YAML

**Симптом:** `⚠ Skipped loading skill(s) due to invalid SKILL.md files`

**Причина:** YAML frontmatter в SKILL.md содержит незаэкранированные спецсимволы (`:`, `"`, `{}`).

**Решение в коде:** `install.sh` генерирует SKILL.md для Codex с description в кавычках.

**Правило:** Всегда оборачивать `description` в кавычки в YAML frontmatter. Проверять валидность после генерации.

---

## 4. Скиллы противоречат друг другу

**Симптом:** Один скилл говорит "всегда сохранять", другой — "спросить перед сохранением". Или один определяет формат отчёта иначе, чем другой ожидает.

**Причина:** Скиллы писались и правились в разное время, без перекрёстной проверки.

**Решение в коде:**
- Общие концепции вынесены в `templates/_shared/` — единое определение, включаемое везде
- Правило в `.claude/rules/skill-authoring.md`: "When adding a new constraint, grep all other skills for conflicting constraints"
- Периодический аудит: `grep` по всем шаблонам на конфликтующие инструкции

**Чеклист при изменении скилла:**
- [ ] Проверить `_shared/` — не затрагивает ли правка shared-контракт?
- [ ] `grep` по другим шаблонам на тему изменяемого поведения
- [ ] Если скилл передаёт данные другому (review → fix), проверить совместимость форматов

## 5. Порядок findings в cp-fix ≠ порядку cp-review

**Симптом:** Фиксер обрабатывает findings в произвольном порядке, а не в порядке отчёта ревьюера (critical → important → minor).

**Причина:** Фиксер переупорядочивает findings по своей логике вместо того чтобы следовать порядку из отчёта.

**Решение в коде:** cp-fix обязан обрабатывать findings строго в том порядке, в котором они указаны в отчёте cp-review. Порядок отчёта = порядок исправления.

**Правило:** Формат вывода скилла A должен быть входом скилла B без перестановки. Закреплено в `.claude/rules/skill-authoring.md` — "Anti-contradiction rules".

## 6. Ad hoc режим — сохранение файлов без спроса

**Симптом:** При запуске `/cp-review` или `/cp-fix` вне основного workflow (ad hoc) скилл автоматически сохраняет отчёт в файл, хотя должен спросить.

**Причина:** Скиллы изначально были написаны только для workflow-режима, где сохранение обязательно. Ad hoc режим добавлен позже, но gate на сохранение не везде проставлен.

**Решение в коде:** В шаблонах cp-review и cp-fix есть явная проверка: если нет активного workflow (нет `.ai/tasks/*/workflow.md`), перед сохранением спросить пользователя. Три варианта: сохранить в файл, передать в cp-fix, показать только на экране.

**Правило:** В ad hoc режиме НИКОГДА не сохранять артефакты без явного подтверждения. В workflow-режиме сохранение обязательно.

## 7. Фиксер не отмечает прогресс в отчёте

**Симптом:** При остановке и продолжении cp-fix непонятно какие пункты уже исправлены. Начинает сначала.

**Причина:** Не было механизма инкрементальной отметки.

**Решение в коде:** cp-fix после каждого исправленного или пропущенного пункта сразу отмечает его в отчёте (✅ fixed / ⏭ skipped / 🔜 deferred). При продолжении — смотрит на отметки и продолжает с первого неотмеченного.

**Правило:** Любой скилл с пошаговой обработкой списка ДОЛЖЕН сохранять прогресс после каждого шага, чтобы можно было продолжить после прерывания.

## 8. Потеря контекста при прерывании сессии

**Симптом:** После прерывания сессии (лимит контекста, ручная остановка) и продолжения — скилл начинает сначала или теряет состояние.

**Причина:** Состояние хранилось только в контексте разговора, а не в файлах.

**Решение в коде:** Для workflow-режима состояние сохраняется в `.ai/tasks/*/workflow.md`. Activity log фиксирует что запускалось. При продолжении — читать workflow.md и log, продолжить с последнего незавершённого шага.

**Правило:** Любой долгий процесс должен сохранять состояние в файл для восстановления после прерывания.

## 9. Timestamp в именах файлов — хардкод вместо реального времени

**Симптом:** Файлы отчётов получают неправильное время в имени (например, `1200` вместо реального).

**Причина:** LLM "угадывает" время вместо вызова `date`.

**Решение в коде:** Явная инструкция в шаблонах с кросс-платформенными командами: `date +%H%M` (Unix/macOS) или `Get-Date -Format 'HHmm'` (PowerShell/Windows). Аналогично для `mkdir -p` → `New-Item -ItemType Directory -Force`.

**Правило:** Никогда не хардкодить и не угадывать динамические значения (время, дату, версии). Всегда получать через команду.

## 10. Документация не обновляется после изменений

**Симптом:** README, `.ai/docs/` расходятся с реальным поведением после правок скиллов.

**Причина:** Документация рассматривается как отдельная задача "на потом", а не как часть изменения.

**Решение в коде:** Правило в `.claude/rules/skill-authoring.md` — секция "Documentation sync rule". Также в `CLAUDE.md`:
> After any skill or behavior change, check and update `README.md`, `README.ru.md`, and `.ai/docs/` if affected

**Чеклист:**
- [ ] `README.md` и `README.ru.md` — обновлены?
- [ ] `.ai/docs/README.md` — навигация актуальна?
- [ ] Соответствующие файлы в `.ai/docs/domains/` — отражают изменения?
- [ ] `.claude-plugin/plugin.json` — версия, описание актуальны?

## Мета-чеклист: перед любым изменением в templates/

1. [ ] Прочитать related шаблоны, которые могут быть затронуты
2. [ ] Проверить `_shared/` на конфликты
3. [ ] Внести изменения в templates (НЕ в skills)
4. [ ] `./install.sh build`
5. [ ] Проверить сгенерированные `skills/` на корректность
6. [ ] Обновить документацию (README, .ai/docs/)
7. [ ] Обновить версию если нужно (plugin.json, marketplace.json)
8. [ ] Если изменён формат данных между скиллами — проверить совместимость цепочки
